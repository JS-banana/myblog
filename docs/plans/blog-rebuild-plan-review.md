# 博客重建方案审核补充（供 Claude 反馈）

## 结论

当前 `blog-rebuild-plan.md` 的主方向是正确的，不需要推翻：
- `Astro + 双仓 + GitHub Actions 同步 + Vercel` 这条路线可行
- 功能边界整体克制，适合个人技术博客
- 页面结构、内容同步思路、V1/V2 划分已经比较清晰

这份方案当前更像“可开工版本”，但距离“长期更稳的执行方案”还差几块关键补丁。真正建议补强的重点，不是框架选型，而是：

1. AI 元数据治理
2. slug / URL 稳定性
3. 同步链路的幂等性与可观测性
4. SEO 细节
5. 上线验收标准

---

## 建议优先补进 V1 的内容（P0）

### 1. 给 AI 生成的元数据增加人工兜底入口

当前方案默认 `slug`、`description`、`tags` 全由 AI 生成，但没有人工覆盖机制。这个在长期维护上风险较高。

**建议规则：**
- frontmatter 中如果手动写了 `slug` / `description` / `tags`，则优先使用手动值
- 只有字段缺失时，才调用 AI 自动生成

**原因：**
- AI 生成质量不稳定时，作者可以局部修正，而不是改脚本逻辑
- 对高价值文章、SEO 重点文章，可以手工优化摘要和 slug
- 可以避免因为模型波动导致元数据不一致

**建议补充到方案中的表述：**
> `slug`、`description`、`tags` 默认由同步脚本自动生成；若源文章 frontmatter 中显式提供这些字段，则优先使用手动值，不再覆盖。

---

### 2. 调整 slug 缓存策略，避免“正文微调导致 slug 失稳”

方案里现在写的是：按“标题 + 内容 hash”匹配缓存，避免重复调用 AI。

这个思路能节省调用成本，但**不适合作为 slug 稳定性的唯一依据**。因为正文一旦小改，hash 就会变，可能导致重新生成 slug，与“首次生成后固定不变”的目标冲突。

**建议改成：**
- 使用 `source path` 或显式 `source id` 作为主键维护 slug 映射
- slug 首次生成后就固化
- 正文更新不应触发 slug 重算
- 只有文章明确是“新文章”或作者手动改了 `slug`，才允许变化

**更稳妥的缓存思路：**
```json
{
  "blog/koa/koa-onion.md": {
    "slug": "koa-onion-model",
    "title": "Koa之洋葱模型分析"
  }
}
```

**建议补充到方案中的表述：**
> `slug` 缓存建议以源文章路径（或稳定 source id）为主键维护，而不是仅依赖标题/内容 hash。文章正文更新时不应导致 slug 变化。

---

### 3. 明确同步链路的幂等、失败保护和可观测性

当前方案写了“workflow 中加重试逻辑，失败时跳过该文章并报告”，方向对，但还不够具体。

**建议至少补充以下能力：**
- `dry-run`：本地或 CI 可先预演本次会新增/更新/删除哪些文章
- `summary`：同步完成后输出摘要（新增几篇、更新几篇、跳过几篇、失败几篇）
- `failure report`：失败时输出具体文章列表和失败原因
- `atomic write`：AI 调用失败或中途异常时，不写入半成品文件
- `idempotent`：同一次内容重复执行，结果应一致
- `unpublish/delete handling`：源文章取消 `publish: true` 或源文件被删除后，目标仓库如何删除对应产物，要写清楚

**这里建议额外补一条：**
- 维护一个同步清单（manifest）或依据缓存索引反查已生成文件，确保“取消发布”后能自动清理旧文章，而不是越积越多

**建议补充到方案中的表述：**
> 同步脚本需支持 dry-run 和 summary 输出；对失败文章输出清单；在 AI 请求失败或写入中断时不得落地产生半成品；当文章取消发布或源文件删除时，应能清理博客仓库中的对应生成文件。

---

### 4. URL 稳定性不应只考虑“旧博客 -> 新博客”

当前方案已经提到：
- `vercel.json` 配置旧 URL 301 重定向：`/pages/:hash` → 新路径

这个还不够。

**还需要考虑的场景：**
1. 旧 VuePress 链接迁移
2. 未来人工修正 slug
3. 旧 frontmatter 里如果已经有 `permalink`，是否能作为历史链接线索保留下来
4. 某些文章未来从一个目录迁移到另一个目录时，旧地址如何处理

**建议：**
- 增加一个 redirect map，例如：`src/content/.redirects.json`
- 同步脚本在发现 slug 人工变更时，自动生成旧地址到新地址的 301 映射
- 不要在清理旧 frontmatter 时直接丢弃 `permalink` 信息；可以先提取并转成 redirect 线索，再清理

**建议补充到方案中的表述：**
> 除旧 VuePress 路径重定向外，还应维护文章级 redirect map，用于承接未来 slug 修正、目录调整或历史 permalink 迁移，保证外链尽可能不失效。

---

### 5. SEO 需要再补几个最小但关键的细节

当前方案已有：
- OG
- Twitter Card
- JSON-LD
- sitemap
- RSS

还建议补充：

#### 5.1 canonical URL
每篇文章页都应输出 canonical，避免重复路径、预览域名、临时路径干扰收录。

#### 5.2 publishedTime / modifiedTime
- `publishedTime` 直接来自文章 `date`
- `modifiedTime` 建议未来支持 `updated` 字段；V1 可以先预留，不强制所有文章补齐

#### 5.3 OG 图策略
- 提供站点默认 OG 图
- 若文章没有单独 OG 图，则回退到默认图
- 如果后续想扩展，可以再做“标题渲染式 OG 图”，但不必纳入 V1

#### 5.4 Vercel 预览环境 noindex
如果使用 Vercel Preview，建议明确：
- 预览环境页面不应被搜索引擎收录
- 正式生产域名才输出索引友好配置

**建议补充到方案中的表述：**
> SEO 除基础 meta 外，还需明确 canonical 输出规则、文章发布时间/更新时间字段来源，以及 Vercel Preview 环境的 noindex 策略。

---

### 6. 验收阶段建议改成“结构化 checklist”

当前方案的验证项还比较概括，比如“检查所有页面渲染正常”。这个在执行时容易变成口头判断。

**建议改成 checklist：**

#### 页面与路由
- [ ] 首页文章列表正常
- [ ] `/page/2` 等分页可访问
- [ ] `/posts/{year}/{slug}` 正常渲染
- [ ] `/about` 正常访问

#### 内容与同步
- [ ] `publish: true` 的文章可进入博客仓库
- [ ] `publish: false` 或取消发布的文章不会继续保留在线
- [ ] slug 缓存命中后不会因正文微调改变 URL
- [ ] AI 失败时 workflow 能明确报错并终止/跳过，不产生脏文件

#### SEO / Feed
- [ ] `rss.xml` 可访问且内容正确
- [ ] sitemap 正常生成
- [ ] 文章页输出 canonical
- [ ] OG / JSON-LD 正常

#### 体验
- [ ] 暗色模式无闪烁
- [ ] 移动端排版正常
- [ ] Lighthouse 性能/SEO 达到预期

#### 重定向
- [ ] 旧 VuePress URL 可正确 301 到新路径
- [ ] 手动修正 slug 后旧链接仍可访问（若已启用 redirect map）

## 强烈建议预留但不必强上 V1 的内容（P1）

### 1. frontmatter 可选增加 `updated`
用于支持文章更新时间，先作为可选字段预留即可。

### 2. 持久化 `sourcePath` / `sourceId`
这会让同步系统更稳：
- 方便溯源
- 方便清理已删除文章
- 方便生成 redirect
- 方便排查哪篇文章同步失败

### 3. 对生成文件增加“不要手改”标记
既然 `src/content/posts/` 是同步脚本写入，建议生成的 markdown 文件顶部加一条注释，例如：

```md
<!-- Generated by sync-posts.mjs. Do not edit directly. -->
```

这样后续维护时更不容易误改。

### 4. 统一日期时区策略
当前 `date: 2022-04-26` 这种写法很简洁，但渲染和排序时最好明确：
- 统一按哪个时区解释
- 是否只按日期，不按具体时间

避免后续部署环境不同时出现日期偏移。

### 5. 增加本地预检命令
建议后续让同步脚本支持：
- 本地 dry-run
- 本地输出 summary
- 本地验证 frontmatter 是否完整

这样你在 `posts` 仓库改完内容后，不一定每次都要靠 GitHub Actions 才知道有没有问题。

## 已明确的 V1 非目标

这部分建议也直接写进正式方案，防止范围膨胀：

- 站内搜索
- 评论系统
- 标签归档页
- 历史图片资源迁移
- 复杂后台 / 编辑系统
- 访问统计面板

这样 V1 的目标就会更清楚：
**先把“稳定发布 + 极简展示 + 自动同步”跑通。**

---

## 我认为最值得立刻修正的一点

如果只选一个必须马上修的点，我会选：

> **不要把“标题 + 内容 hash”当作 slug 稳定性的核心依据。**

因为这和“文章首次生成 slug 后长期稳定”这个目标天然冲突。正文更新是高频行为，URL 改变应该是低频、受控行为。

## 一句话总结

这份方案已经可以开工，但如果希望它不仅“能上线”，还要“后续少返工”，建议优先补上：

**人工兜底 + slug 稳定策略 + 同步幂等/清理 + redirect map + 更细的 SEO/验收规则。**
