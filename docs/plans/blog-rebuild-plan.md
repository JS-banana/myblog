# 博客重建方案

## Context

用户的旧博客（VuePress 1.x）技术过时，需要重建。当前写作内容在 `posts` 仓库（Obsidian 管理），包含 22 篇技术博客、4 篇公众号文章等。目标是建立一个现代、高性能、极简风格的个人技术博客，通过自动化流程从内容仓库同步文章并部署。

## 核心决策

| 决策项 | 选择 |
|--------|------|
| 仓库 | 双仓：`posts`（内容）+ `JS-banana/blog`（展示，复用旧仓库名） |
| 框架 | Astro（零 JS 默认输出、Content Collections） |
| 部署 | Vercel 为主，域名后续绑定 |
| 内容控制 | Frontmatter `publish: true` 标记制 |
| 视觉风格 | 极简文字流（类 tw93.fun） |
| 样式方案 | 纯 CSS 变量（不用 Tailwind，极简风排版仅需 ~250 行） |
| V1 功能 | 文章列表+分页、RSS、暗色模式、SEO、关于页 |
| 图片托管 | Cloudflare R2（免费额度大，全球 CDN） |
| AI 服务 | 国内大模型（OpenAI SDK 兼容接口），用于生成 slug 和 description |

## 当前阶段边界（local-only）

本计划分成两个阶段，但**当前只执行本地 V1/P0 完成**：

- **当前阶段：local completion**
  - 只完成博客仓库本地可验证的 V1/P0 基线
  - 只落地 receiver-side 合同、页面、SEO、文档和本地验证
  - `src/content/.redirects.json` 在本阶段作为本地 source of truth 和验证依据
  - `src/content/.slug-cache.json`、样例内容、schema 约定只服务于本地可交付
- **后续阶段：launch**
  - 真实域名、部署、GitHub push、远端 `posts` 同步联通
  - 部署侧 301/redirect 具体落地
  - 这些都保持后置，不构成本轮完成条件

`docs/plans/blog-rebuild-plan-review.md` 保持为**评审补充记录**，不在本轮被改写成执行计划。

## 写作时的 Frontmatter（极简）

写文章时只需维护 3 个字段：

```yaml
---
title: 文章标题
date: 2022-04-26
publish: true
---
```

### 元数据生成与人工覆盖规则

`slug`、`description`、`tags` 默认由同步脚本 AI 自动生成。若源文章 frontmatter 中显式提供了这些字段，则优先使用手动值，不再覆盖。

- **默认模式**（零负担）：只写 title/date/publish，其余全自动
- **精细控制模式**（高价值文章）：手写 slug/description/tags 覆盖 AI 生成结果

旧有的 `permalink`、`categories` 等 VuePress 字段不再保留。同步脚本清理时，先提取 `permalink` 作为 redirect 线索，再丢弃。

## 内容同步机制

### 当前阶段：receiver-side contract（本地真源）

当前阶段只定义和验证博客仓库这一侧的最小合同，不实现完整远端同步链路：

- `src/content/.slug-cache.json`：slug 的稳定映射来源
- `src/content/.redirects.json`：本阶段的 redirect source of truth
- `src/content.config.ts`：只为本地 receiver contract 提供必要 schema；可预留 `updated` / `sourceId`，但不把它们扩展成远端同步实现
- `src/content/posts/`：放置同步后生成的样例/产物，用于本地证明合同成立
- `docs/plans/blog-rebuild-v1-checklist.md`：本阶段执行与验收的结构化清单

当前阶段的目标是：**本地能验证、合同能承接、边界能说清**。
不在本阶段做的事：
- 不做真实 `posts` 仓库 GitHub Actions 联通
- 不做生产域名或最终 `site` 值定稿
- 不做 Vercel / deploy runtime 的 redirect 落地
- 不做 remote sync engine 或 generalized pipeline

### 流程

```
posts 仓库 push → GitHub Actions → 扫描 publish:true
  → AI 生成 slug/description/tags（手动值优先）
  → 转换内容 → 清理取消发布的旧文章
  → push 到博客仓库 → Vercel 自动部署
```

> 上面的流程属于 **launch 阶段** 目标；当前阶段只要求把 receiver-side 合同、样例内容和本地验证准备好。

### 同步脚本（posts/.github/scripts/sync-posts.mjs）

核心职责：
1. 递归扫描 `blog/`、`wechat/`、`me/`、`year-in-review/`、`story/` 目录
2. 过滤 `publish: true` 的文章
3. 检查 frontmatter 是否已手动提供 slug/description/tags，缺失的才调用 AI 生成
4. 清理旧 VuePress 语法（`<!-- more -->`、`:::tip` 容器）和无用 frontmatter 字段
5. 按年份子目录输出到博客仓库 `src/content/posts/2022/koa-onion-model.md`
6. 清理博客仓库中已取消发布或源文件已删除的旧文章
7. 输出同步摘要（新增/更新/删除/跳过/失败）

生成的 markdown 文件顶部加注释标记：
```md
<!-- Generated by sync-posts.mjs. Do not edit directly. -->
```

### Slug 稳定性与缓存策略

采用**日期+语义化短名**格式：`/posts/2022/koa-onion-model`。

**缓存以源文件路径为 key**（而非标题/内容 hash），slug 首次生成后固化，正文更新不触发 slug 重算：

```json
// src/content/.slug-cache.json
{
  "blog/2022/Koa之洋葱模型分析.md": {
    "slug": "koa-onion-model",
    "title": "Koa之洋葱模型分析",
    "year": "2022"
  }
}
```

- 新文章（缓存中不存在的路径）：调用 AI 生成 slug，写入缓存
- 已有文章（缓存命中）：直接使用缓存 slug，不管正文是否变更
- 手动覆盖：frontmatter 中显式写了 `slug`，以手动值为准，同时更新缓存
- 同一年份 slug 重复时自动追加 `-2` 后缀

### Redirect Map 与历史链接迁移

本阶段维护 `src/content/.redirects.json` 作为本地 source of truth。它用于：
- 承接未来 slug 修正、目录调整、历史 permalink 迁移
- 让本地检查和样例内容能证明 redirect 关系成立
- 作为后续 launch 阶段生成部署侧 redirect 的输入

本阶段**不要求**构建时自动生成最终线上重定向规则；那个能力留到 launch 阶段再决定实现路径。

```json
{
  "/pages/b92fbe/": "/posts/2022/koa-onion-model",
  "/posts/2022/old-slug": "/posts/2022/new-slug"
}
```

来源：
- 旧 VuePress `permalink` 字段：同步脚本清理时自动提取并写入
- Slug 变更：手动修正 slug 时，脚本自动将旧 slug 路径追加到 redirects
- 当前阶段仅验证该映射文件的存在、格式和用法；部署侧消费方式留待 launch 阶段决定

> 说明：在当前阶段，`src/content/.redirects.json` 只作为**本地源真值**和文档/验证依据，不等于部署侧 redirect 已经完成。部署侧规则与真实 301 证明留待后续 launch 阶段。

### 同步脚本的幂等性、失败保护与可观测性

- **dry-run**：支持 `--dry-run` 参数，预演本次会新增/更新/删除哪些文章，不实际写入
- **summary**：每次执行完输出摘要日志（新增 N 篇、更新 N 篇、删除 N 篇、跳过 N 篇、失败 N 篇）
- **failure report**：AI 调用失败或写入异常时，输出具体文章路径和错误原因
- **atomic write**：AI 请求失败或中途异常时，不写入半成品文件；先写临时文件，成功后再 rename
- **idempotent**：相同输入重复执行结果一致
- **unpublish/delete**：同步脚本对比缓存索引与当前 publish:true 文章列表，删除博客仓库中已不存在于源的文章

### AI 生成逻辑

- **服务**：国内大模型，通过 OpenAI SDK 兼容接口调用
- **Secret**：`AI_API_KEY` + `AI_BASE_URL` 存到 GitHub Secrets
- **调用方式**：批量处理，单次请求传入标题+正文前 500 字，返回 JSON：
  ```json
  {
    "slug": "koa-onion-model",
    "description": "深入分析 Koa 框架的洋葱模型原理...",
    "tags": ["Koa", "Node.js", "中间件"]
  }
  ```
- **重试**：AI 调用失败自动重试 2 次，仍失败则跳过并报告

### Workflow（posts/.github/workflows/sync-to-blog.yml）

- 触发：push to main，paths 限定内容目录
- 步骤：checkout 两个仓库 → 安装依赖 → 执行同步脚本 → commit & push 到博客仓库
- Secrets：`BLOG_REPO_TOKEN`（PAT）、`AI_API_KEY`、`AI_BASE_URL`

## Content Collection Schema（src/content.config.ts）

```typescript
import { defineCollection } from 'astro:content';
import { glob } from 'astro/loaders';
import { z } from 'astro/zod';

const posts = defineCollection({
  loader: glob({ pattern: '**/*.md', base: './src/content/posts' }),
  schema: z.object({
    title: z.string(),
    date: z.coerce.date(),
    description: z.string().default(''),
    tags: z.array(z.string()).default([]),
    slug: z.string(),
    source: z.enum(['blog', 'wechat', 'year-in-review', 'me', 'story']).optional(),
    sourcePath: z.string().optional(),
  }),
});

export const collections = { posts };
```

## 暗色模式

- CSS 变量体系：`:root`（亮色）+ `[data-theme="dark"]`（暗色）
- `<head>` 内联脚本读取 localStorage / prefers-color-scheme，防止闪烁
- ThemeToggle 组件：纯 Astro，无框架依赖

## SEO

- Head 组件统一管理：Open Graph、Twitter Card、JSON-LD (BlogPosting)
- **canonical URL**：每篇文章页输出 canonical，避免预览域名/重复路径干扰收录
- **publishedTime**：来自文章 `date` 字段，输出为 `article:published_time`
- **OG 图**：站点默认 OG 图（`public/og-default.svg`），文章无单独配图时回退到默认
- **Vercel Preview noindex**：非生产域名自动输出 `<meta name="robots" content="noindex">`
- `@astrojs/sitemap` 自动生成 sitemap
- `@astrojs/rss` 生成 RSS feed
- AI 生成的 description 自动用于 meta description 和 OG description

## 图片方案

**Cloudflare R2** 作为统一图床：
- 免费额度：10GB 存储 + 每月 1000 万次读取
- 自带 CDN，国内外访问稳定
- 绑定自定义域名（如 `img.yourdomain.com`）
- 上传方式：写个小脚本或用 rclone 等工具

现有文章的图片处理：
| 来源 | 当前状态 | V1 策略 |
|------|---------|---------|
| blog/ | jsdelivr CDN 外链 | 保持原样，后续可迁移到 R2 |
| wechat/ | 微信图床 mmbiz | 保持原样，观察防盗链 |
| 新文章 | — | 统一上传到 R2 |
